1MCP とは何か

Model Context Protocol は、LLM (Host) ↔ ツール/データ提供者 (Server) を繋ぐオープンプロトコル。「LLM 業界の USB-C」と呼ばれる。

1.1 3層アーキテクチャ

MCP は Host / Client / Server の 3 者で構成される。Host が AI アプリ本体 (Claude Desktop, ChatGPT, IDE 等)、Client は Host 内で各サーバーへの接続を担う薄い層、Server がツール・データ・プロンプトを公開する独立プロセス。

[Host: Claude.ai / ChatGPT / Grok / Copilot] ├─ MCP Client 1 ─── HTTPS ──── Server A (社内 CRM API) ├─ MCP Client 2 ─── HTTPS ──── Server B (社内ナレッジ DB) └─ MCP Client 3 ─── HTTPS ──── Server C (Salesforce / Notion / GitHub 等)

1.2 Primitives (サーバーが提供できる 3 種類)

🔧 Tools
AI が 実行できる関数。API 呼び出し・DB クエリ・ファイル操作など。tools/list / tools/call。入力は JSON Schema で型定義。書き込み系は destructiveHint: true 必須。
📚 Resources
AI が 読み込むデータ。ファイル内容・DB レコード・API レスポンスなど。resources/list / resources/read。URI でアドレッシング。
💬 Prompts
ユーザーが呼び出せる 再利用可能テンプレート。スラッシュコマンド等で展開。「議事録を要約して」「インシデント報告書を作って」のような業務テンプレ向き。

1.3 プロトコルバージョンと Transport

バージョン状態主な変更

2024-11-05Legacy初期仕様 (HTTP+SSE)、互換目的のみ

Transport は 2 種類:

  • STDIO — クライアントがサーバーをサブプロセス起動。ローカル MCP 専用 (Claude Desktop でローカルファイルを読むなど)
  • Streamable HTTP — 単一 URL (例 https://api.example.com/mcp) が POST/GET を受ける。Web UI コネクタとして繋ぐ場合はこちら一択
💡
覚えるべきは 1 つだけ: 自社サービスを Claude.ai / ChatGPT / Grok の Web UI に繋ぐなら、必ず Streamable HTTP ベースの Remote MCP Server を実装する。STDIO はローカルアプリ用なので Web UI からは届かない。

2各 Web UI の対応状況

2026 年 5 月時点、主要 4 UI すべてが MCP コネクタを受け入れる。ただし対応プラン・審査・配信方法・認証の柔軟性に差がある。

2.1 Claude.ai (Anthropic) — 本家 / 最も成熟

  • 2025 年 5 月にベータ提供開始。Free でも 1 個利用可、Pro/Max/Team/Enterprise で複数追加可
  • 個人での追加: Settings → Connectors → + Add custom connector に URL を貼る
  • 組織配信: Owner が Organization settings → Connectors に登録 → 全メンバーに表示
  • Connector Directory: Anthropic 公式の申請窓口経由で受付。手動レビュー約 2 週間。tool annotation (title, readOnlyHint, destructiveHint) の不足が却下原因の約 30%
  • 375+ コネクタがディレクトリに掲載済み

2.2 ChatGPT (OpenAI) — Developer Mode + Apps SDK

  • 2 つのレイヤー: Developer Mode (個人/組織向けカスタム MCP) と Apps SDK (Directory 公開)
  • Plus/Pro でベータ利用可。Business/Enterprise/Edu は admin 経由で RBAC 制御
  • 2025-12-17 から Apps Directory submission 受付開始、2026 初頭から段階的公開
  • カスタム API キー / mTLS は明示的に非対応。OAuth 2.1 + PKCE / CIMD / DCR のみ
  • 審査要件: MFA/SMS 認証不可のデモアカウント、Privacy Policy 必須、CSP 必須

2.3 Grok (xAI) — 2026/5 GA、後発だが揃っている

  • 2026 年 5 月 6 日に Connectors 正式公開 (Web/iOS/Android)
  • "Bring Your Own MCP" でカスタム MCP URL を直接追加可
  • Built-in: GitHub, Notion, Linear, Google Workspace, Microsoft 365
  • 有料プラン限定 (Free tier は connector メニュー非表示)。具体的プラン区分は 未確認
  • Connector Directory / 公開審査の仕組みは未整備 未確認

2.4 Microsoft Copilot — エンタープライズ志向、ハードル高

  • Copilot Studio + M365 Copilot 双方で MCP サポート
  • M365 Copilot フルライセンス必須(2025-11-19 以降、約 $30/月/ユーザー)、Frontier プログラム enrollment 前提
  • Entra ID テナント前提の認証、Admin Center → Copilot → Frontier ON で有効化
  • 個人向け Copilot Pro/Free での MCP は限定的 未確認

3認証アーキテクチャ

MCP の認証は OAuth 2.1 + 4 つの RFC を組み合わせる。鍵は「自前で OAuth サーバーを書かなくていい」設計。

3.1 必須 RFC 群

RFC役割MCP での要件
RFC 9728Protected Resource MetadataMCP server MUST。/.well-known/oauth-protected-resource で authorization_servers を公開
RFC 8414Authorization Server MetadataAS MUST。/.well-known/oauth-authorization-server
RFC 7591Dynamic Client Registration (DCR)Client/AS SHOULD。Claude.ai が URL 貼るだけで繋がる仕組みの根幹
RFC 8707Resource IndicatorsClient MUST。resource パラメータで token の audience を縛り、Confused Deputy 攻撃を防ぐ

3.2 ロール分離 (重要)

MCP は OAuth 2.1 の 3 ロールを厳密に踏襲する:

🛡 Resource Server
= MCP server 本体。Bearer token を検証してリソースを返すだけ。OAuth サーバーを兼ねる必要なし
📱 Client
= Claude.ai / ChatGPT。ユーザーの代理でリクエスト。DCR でクライアント登録
🔐 Authorization Server
外部委譲可能。Auth0 / WorkOS / Stytch / Cloudflare / Keycloak 等のホスト型 IdP を利用できる
✅
これが MCP 認証設計の最大の利点: MCP サーバーは Resource Server だけを担えばよく、Authorization Server は外部の既存 OAuth プロバイダ (Auth0/WorkOS/Cloudflare 等) に委譲できる。自前で OAuth サーバーを書く必要がない。

3.4 API Key で繋ぎたい場合

⚠️

Web UI コネクタとして API Key / 静的 Bearer Token で繋ぐのは事実上不可:

  • ChatGPT: 公式に「カスタム API キー / mTLS は非対応」と明言
  • Claude.ai: UI から任意 Bearer ヘッダ指定不可。OAuth に見せかける薄いラッパーが必要
  • Claude API (MCP Connector): SDK 経由ならアプリ側が事前取得した access token を直接渡せる

つまり 「OAuth 2.1 を立てる」が事実上の必須要件。ただし AS は既製品で OK。

4.2 必須インフラ要件

  • HTTPS 公開 URL (Anthropic/OpenAI の IP から到達可能)。VPN 内不可
  • Streamable HTTP transport: 単一 URL で POST + GET。Accept: application/json, text/event-stream を受ける
  • MCP-Protocol-Version ヘッダ検証 (不一致は 400)
  • Origin ヘッダ検証 (DNS rebinding 攻撃対策)
  • /.well-known/oauth-protected-resource でメタデータ配信
  • 401 時の WWW-Authenticate ヘッダ: Bearer resource_metadata="..."
  • セッション管理: Mcp-Session-Id ヘッダで識別。stateless でも動くが elicitation/sampling/進捗通知 を使うなら stateful 化

4.3 実装 5 ステップ (推奨経路: Cloudflare Workers + GitHub OAuth テンプレ)

  1. テンプレからプロジェクト作成
    npm create cloudflare@latest -- --template=cloudflare/ai/demos/remote-mcp-github-oauth
    OAuth プロバイダ + MCP server がワンセットで雛形に入っている
  2. Tool を実装する
    McpAgent を継承し、this.server.tool(name, schema, handler) で社内 API を 3〜5 個ラップ。必ず title / readOnlyHint / destructiveHint を annotation として付ける (Claude Directory 審査の最大の落とし穴)
  3. OAuth と結線
    workers-oauth-provider で AS を内包させるか、Auth0/WorkOS を外付け。/.well-known/oauth-protected-resource で authorization_servers を公開
  4. ローカルで動作確認
    npx @modelcontextprotocol/inspector でブラウザ UI を起動 → Streamable HTTP で http://localhost:3000/mcp を叩いて tools が見えるか確認
  5. デプロイして Claude.ai に登録
    wrangler deploy → 公開 URL を Claude.ai → Settings → Connectors → + Add custom connector に貼り付け。OAuth が走ってツールが使えるようになる

サンプルコード (TypeScript / Cloudflare Workers)
app.ts
import { McpAgent } from "agents/mcp";
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";

export class MyMCP extends McpAgent {
  server = new McpServer({ name: "Internal CRM", version: "1.0.0" });

  async init() {
    this.server.tool(
      "search_customers",
      "顧客名や ID で社内 CRM を検索する",
      { query: z.string() },
      async ({ query }) => {
        const r = await fetch(`https://crm.internal/api/search?q=${query}`, {
          headers: { Authorization: `Bearer ${this.props.accessToken}` }
        });
        return { content: [{ type: "text", text: await r.text() }] };
      },
      { annotations: { title: "顧客検索", readOnlyHint: true } }
    );
  }
}

export default new OAuthProvider({
  apiHandler: MyMCP.serve("/mcp"),
  defaultHandler: GitHubHandler,
  authorizeEndpoint: "/authorize",
  tokenEndpoint: "/token",
  clientRegistrationEndpoint: "/register",
});

5セキュリティと運用

攻撃面が増えるため、Confused Deputy / Token Passthrough / DNS Rebinding 等の対策が必須。

5.1 必須セキュリティ対策

🎯 Audience Binding
RFC 8707 の resource を必ず指定。RS は token の aud claim が自分宛か検証する MUST
🔑 PKCE S256
パブリック・コンフィデンシャル問わず 必須。認可コード横取り攻撃の防止
⏱ 短命トークン
Access token 1 時間 + Refresh rotation (Notion 例)。Public client は rotation MUST
🚫 Token Passthrough 禁止
受信した token を下流 API にそのまま転送 MUST NOT。MCP server が別途 OAuth client として新 token を取得する
🛡 Per-Client Consent
DCR で攻撃者が悪意ある redirect_uri を登録できないよう、__Host- プレフィックス + client_id バインドの consent cookie
🌐 Origin 検証
DNS rebinding 対策で Origin ヘッダ検証必須。ローカル時は 127.0.0.1 bind

5.2 運用視点のチェックリスト

  • レート制限: Cloudflare Rate Limiting / Vercel Firewall / API Gateway Usage Plan
  • 監査ログ: (timestamp, user_id, tool_name, args hash, latency, status) を構造化ログ出力
  • スコープ最小化: tool ごとに scope を割り、withMcpAuth の requiredScopes で deny-by-default
  • マルチテナント: 1 ユーザー = 1 OAuth grant。Durable Object や DB 行をユーザー単位に分割
  • テストアカウント: MFA/SMS 認証なしで用意 (Claude/ChatGPT の Directory 審査要件)
  • CSP 設定: ChatGPT Apps SDK 審査では fetch 元ドメイン制限が必須
🚨
最大のリスク: Confused Deputy 攻撃 — MCP server が下流 API の OAuth ブリッジになる場合、static client_id + DCR + 同意 cookie の組み合わせで攻撃が成立する。MCP server 自身で per-client consent ページを表示し、第三者 AS にフォワードする前に承認を取ること。

7.1 アーキテクチャ推奨パターン

✨ Recommendation

2 段構えで進めるのが安全:

  1. P1-P2 を 1 〜 2 週間で完了: Cloudflare Workers + workers-oauth-provider で主要社内システム (Wiki / Ticketing / DWH / CRM 等) のうち 1-2 個を MCP 化。Claude.ai の Pro/Max ユーザー向けに「自分で URL を貼って繋ぐ」運用で配布
  2. P3 を 1.5-2 ヶ月で完了: 監査ログと組織配信を実装。Claude Team / Enterprise の Owner が組織レベルで配信できる状態にする

7.2 既存資産との接続パターン

既存資産MCP コネクタ化での扱い
Bedrock AgentCore (社内 AI エージェント基盤)AgentCore Runtime は MCP Protocol を正式サポートするため、既存エージェントを MCP server として再公開可能。ただし URL が AWS 固定形式なので Web UI 接続には CloudFront 等を前段配置
VPC / プライベートリンク 内のサービスWeb UI からは public internet 経由のみ到達可。VPC 内 MCP は不可。Cloudflare Tunnel か Lambda Function URL で公開エンドポイント化が必要
SaaS 連携 (CRM / 業務システム)SaaS のデータを露出する MCP server を別途立てる。SaaS 側は OAuth Client として動作 (Token Passthrough は禁止、新トークンを再取得する)
IAM ポリシー (アクセスキー方式を避けたい)OAuth 2.1 + DCR は IAM アクセスキー方式と無関係。Auth0 / WorkOS のような IdP 委譲なら長期キーを発行せず短命トークンで運用可能

7.3 配信戦略の選択肢

  • 社内利用が主目的: Claude Team / Enterprise Owner が組織コネクタとして配信 → メンバーは自分のアカウントで OAuth 認証して有効化
  • 社外パートナーにも開放したい: Connector Directory への申請 + ChatGPT Apps Directory 申請の二本立て
  • 既存 AI エージェント基盤がある場合: 既存基盤を優先しつつ、Web UI コネクタ化は P1 (最小 PoC) だけ間に合わせて並行で進めるのが現実的
📌
意思決定が必要な論点:
  1. OAuth Authorization Server を どこに置くか (Cloudflare 同居 / Auth0 / WorkOS / Keycloak)
  2. 最初に MCP 化する社内システムをどれにするか (使用頻度 × 機密度 × 既存 API 整備度で評価)
  3. Web UI コネクタを 個人配布 で始めるか Team 組織配信 から始めるか (有料プラン契約の必要性)
  4. OAuth プロバイダ選定の基準: 既存 IdP 連携 / SSO 要件 / コスト / DCR 対応の有無
